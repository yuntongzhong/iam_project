# AI 辅助编程说明

## 目标拆解

- 先从赛题原文抽取明确功能边界和评分项
- 将需求拆为工程初始化、数据建模、协议实现、安全增强、管理台、Demo 客户端、文档七个阶段
- 对 OAuth2 / PKCE / Spring Authorization Server 的关键实现细节先做核对，再写代码
- 先搭建可编译骨架，再通过 `gradlew test` 反向驱动修正库 API 和配置问题

## AI 贡献点

- 生成多模块 Gradle 结构与 Spring Boot 配置
- 生成实体类、Repository、Service、Controller、Security 配置
- 生成 TOTP、异常登录检测、审计脱敏、管理台页面
- 生成 Demo 客户端与答辩文档草案

## 关键 Prompt / 指令类型

- “按题意拆评分项，列出协议合规、权限模型、SSO、工程质量缺口”
- “基于 Spring Authorization Server 实现授权码 + PKCE + OIDC”
- “把部门角色继承映射成 Spring Authorities，并输出用户最终生效权限”
- “为 admin.html 生成投屏友好的治理控制台布局，并加入错误反馈”
- “补齐单点退出链路、首次 MFA 绑定后回跳、403 友好页”
- “补写测试用例，验证 TOTP、锁定策略、回跳恢复、批量导入、令牌 TTL”

## 人工决策点

- 明确用 `Department -> Role` 表达继承，而不是在 SQL 层硬编码权限展开
- 确定客户端采用显式端点配置，避免测试阶段依赖 OIDC 发现
- 保持外部 MySQL 8 为正式运行方案，同时用 H2 支撑测试

## 本轮补漏的人机分工

- AI 负责：
  - 生成 CSV 批量导入服务、DTO、管理台预检界面
  - 生成 OAuth2 客户端令牌策略字段、JDBC 授权持久化接线
  - 生成审计目标识别、JWK 文件化加载、提交打包脚本与文档初稿
  - 生成和修正针对高风险链路的自动化测试
- 人工负责：
  - 确认赛题理解与最终交付目标
  - 决定批量导入只做 CSV，不扩展 Excel
  - 决定交付物中必须包含答辩包，而不是只交代码
  - 最终录制演示视频并替换占位文件

## 返工与修正记录

- 发现 `start-services.cmd` 的真实问题不是脚本本身，而是缺少 MySQL 前置检查，因此补了数据库可达性检测。
- 发现退出中转页使用 `String.formatted` 时被 CSS 中 `%` 误伤，导致退出页面渲染异常，因此重写了退出结果链路。
- 发现客户端此前只展示了过期时间，但服务端并未显式持久化 Token TTL，因此补上了客户端级令牌策略。
- 发现审计只能说明“谁在什么时间做了什么”，但不能明确“登录什么系统”，因此增加目标系统识别与展示。
- 发现赛题要求支持批量导入，而现有实现仅支持手工创建，因此新增了 CSV 模板、预检和整批提交。

## 验收步骤

1. `.\gradlew.bat test`
2. `scripts\package.cmd`
3. `scripts\start-services.cmd`
4. 控制台执行一次 CSV 预检与正式导入
5. App A 完整登录并绑定 TOTP
6. App B 复用 IAM 会话完成 SSO
7. 连续输错密码 5 次，展示锁定与审计
8. `scripts\package-submission.cmd -AuthorName <姓名>`

## 需要保留的 AI 使用证据

- 需求拆解与评分映射对话截图
- 协议实现与安全设计讨论记录
- 前端控制台设计优化记录
- 自动化测试补写与失败修复记录
- 最终提交打包脚本生成过程记录

## 最终结论

- AI 主要承担了需求拆解、骨架生成、局部实现、文档起草与测试补齐的加速角色。
- 关键安全策略、交付边界和答辩表达方式由人工拍板。
- 最终结果不是“直接生成即提交”，而是“AI 生成 + 人工校准 + 测试回归 + 交付封装”的联合产物。
